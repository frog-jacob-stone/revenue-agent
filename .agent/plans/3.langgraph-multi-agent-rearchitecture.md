# LangGraph Multi-Agent Re-architecture

🔴 **Complex** — Multi-week, foundational rewrite. Must be broken into sub-plans before executing. This document is the master plan that scopes the work and sequences sub-plans.

## Context

The current orchestrator and `actions` table grew organically. They work, but the abstractions blur three distinct responsibilities into one table and one step-class hierarchy. Recent conversations surfaced two converging insights:

1. **The `actions` table is overloaded.** It's simultaneously the approval queue, the workflow state-reconstruction record, and the granular audit trail. `step_kind` is the duct tape that lets the inbox filter to "rows humans care about." If approvals were their own table and audit_log captured per-step events, `step_kind` would dissolve along with most of the step-class hierarchy.

2. **The bespoke orchestrator is paying maintenance cost we don't have to pay.** Resume on server restart doesn't survive (acknowledged in `docs/ARCHITECTURE.md`). Conditional flow is bolted onto step instances via `skip_if`. Critique loops are special-cased framework logic. LangGraph solves all three with off-the-shelf primitives — checkpointer, conditional edges, graph cycles.

The PRD says "no LLM frameworks." That constraint is being relaxed for this rewrite because the long-term vision needs primitives we don't want to hand-roll: sub-agents, agent-to-agent messaging, parallel agent execution, supervisor patterns, time-travel debugging.

**The intended outcome:** an architecture that scales cleanly to a team of agents — agents critiquing each other, sub-agents being spawned mid-workflow, supervisor agents routing to specialists, mixed supervised/unsupervised workflows, all auditable. Each capability becomes a normal LangGraph idiom, not a custom framework feature.

## The new architecture

### Core abstractions

| Concept | Today | New |
|---|---|---|
| **Workflow** | `workflows` table + `actions` rows reconstruct state | `workflows` row + LangGraph checkpointer holds in-flight state |
| **Graph** | `Chain(kind, pattern, agent_slug, steps)` | `StateGraph` per workflow kind, registered by kind |
| **Step** | 5 step classes (`LLMStep`, `TaskStep`, `CritiqueStep`, `CheckpointStep`, `ExecutionStep`) | Async functions registered as graph nodes. No class hierarchy. |
| **Pause for approval** | `CheckpointStep` / `ExecutionStep` write `actions` rows in `proposed` status | `interrupt_before=True` on a node + `approvals` row written by hook |
| **Conditional flow** | `skip_if` predicate on next step | `add_conditional_edges(from_node, router_fn, mapping)` |
| **Critique loop** | `CritiqueStep` with `critiques_step_index`, `max_attempts`, `retry_of_action_id` chains | Graph cycle: critique node has conditional edge back to draft node when failed; attempt count lives in graph state |
| **Approval queue** | `actions WHERE status='proposed'` filtered by `step_kind IN ('checkpoint','execution')` | `approvals WHERE status='pending'` — no filter logic |
| **Audit trail** | `audit_log` (sparse) + `actions` rows (dense) | `audit_log` only. Every node entry/exit/failure emits an event. |
| **Agent** | Class with `system_prompt`, `allowed_tools` | Same — but invocable from any graph node, sub-agent, or chat surface via `invoke_agent(slug, input, ctx)` |
| **Agent message** | n/a | New `agent_messages` table for agent-to-agent communication threading |
| **Sub-workflow** | n/a (one chain per workflow today) | A node spawns a child workflow with `parent_workflow_id`; nested traces in UI |

### Data model

**Tables that change:**

- **`workflows`** — keep. Add `parent_workflow_id uuid null` (for sub-workflows) and `state jsonb` (or rely on LangGraph's checkpointer tables). Drop `pattern` and `current_step` columns (graph state replaces them).
- **`approvals`** — new. Lifecycle-only. Schema:
  ```
  id uuid PK
  workflow_id uuid FK → workflows.id
  node_name text                       -- which graph node requested approval
  agent_slug text                      -- the agent acting (for ACL / display)
  action_type text                     -- send_email, write_rev_rec, etc.
  proposed_payload jsonb
  executed_payload jsonb               -- the human's edits, if any
  status text CHECK IN (pending, approved, rejected, executed, failed)
  risk_level text
  summary text
  reasoning text
  approved_by text
  approved_at timestamptz
  rejected_by text
  rejection_reason text
  executed_at timestamptz
  error text
  created_at timestamptz
  ```
- **`actions`** — drop after migration. (Keep an archival snapshot if there is any prod data we care about.)
- **`audit_log`** — keep. Expand event taxonomy to capture per-node entry/exit, agent invocations, sub-workflow spawn, etc.
- **`agent_messages`** — new. For agent-to-agent communication. Schema:
  ```
  id uuid PK
  workflow_id uuid FK
  from_agent_slug text
  to_agent_slug text
  parent_message_id uuid null FK → agent_messages.id  -- threading
  content jsonb
  created_at timestamptz
  ```
  Phase 4 deliverable; not needed for the initial migration.

**Tables unchanged:** `agents`, `memories`, integrations stay as-is.

### Framework choices

- **`langgraph`** — graph engine
- **`langgraph-checkpoint-postgres`** — persistent checkpointer using our existing Supabase Postgres
- **Async-first** — every node is `async def`; matches our codebase
- **No LangChain dependency** — LangGraph works with raw `openai` / `anthropic` SDKs, which we already use

### How the long-term vision maps to this

Each capability you named becomes a natural LangGraph idiom:

| Vision | LangGraph idiom |
|---|---|
| Agents communicating | A node invokes an agent and writes to `agent_messages`; another node reads the thread |
| Agents critiquing each other | Critic node invokes a critic agent; conditional edge routes back to draft on fail. Same pattern works for any "X reviews Y" |
| Workflows containing agents | Already the model — every node can invoke an agent |
| Some supervised, some not | A supervised node has `interrupt_before=True`; an unsupervised node doesn't. Mix freely in one graph. |
| Agents spinning up sub-agents | A node calls `await spawn_workflow(kind, input, parent_workflow_id=ctx.workflow_id)`. Audit captures the parent/child link. UI renders nested traces. |
| Supervisor agent routing to specialists | LangGraph's documented "supervisor" pattern: supervisor node + conditional edges to specialist nodes. Idiomatic. |
| Parallel agent execution | LangGraph's `Send` API: a node can dispatch to N other nodes in parallel and collect their results. |

None of these need framework-level additions in the new world. They're all node + edge declarations in a `StateGraph`.

## Phasing

Six phases. Each is its own sub-plan that gets written before that phase starts. The phases are ordered so that **at every phase boundary, the system is shippable and tests pass.**

### Phase 0 — Foundations (~1 week)

Goal: install the new infrastructure without removing anything. Both orchestrators coexist.

- Add `langgraph` and `langgraph-checkpoint-postgres` to `pyproject.toml`
- Migration `0010_create_approvals_table.sql` — new `approvals` table, indexes, RLS off
- Migration `0011_workflows_parent_id.sql` — add `parent_workflow_id` column to `workflows`
- Build `app/orchestrator_v2/` (parallel to `app/orchestrator/`) containing:
  - `runner.py` — wraps LangGraph: registers graphs, drives execution, writes audit events on node entry/exit, surfaces approvals
  - `approval.py` — service layer: `propose_approval(...)`, `approve(...)`, `reject(...)`, `mark_executed(...)`. Each writes audit events.
  - `state.py` — base TypedDict patterns for graph state
  - `agent_invoke.py` — `invoke_agent(slug, input, ctx)` — uniform entry point usable from any node, chat, or sub-agent context
  - `spawn.py` — `spawn_workflow(kind, input, parent_workflow_id)` for sub-workflows
- Build `app/services/approvals.py` — DB layer for the new table
- Build `app/routers/approvals.py` — `GET /approvals`, `POST /approvals/{id}/approve`, `POST /approvals/{id}/reject`
- Tests for the runner, approval service, and the agent_invoke primitive (no chains migrated yet)

**Exit criteria:** `pytest` passes; new tables exist; old orchestrator and `actions` flow untouched.

**Sub-plan to write before starting:** `4.path-b-phase-0-foundations.md`

### Phase 1 — Migrate `content_publish` (simplest chain) (~1 week)

Goal: prove the pattern with the smallest, lowest-risk chain.

- Define `app/orchestrator_v2/graphs/content_publish.py` as a `StateGraph`
- Two nodes: `propose_post` (auto), `post_to_linkedin` (interrupt_before, executes via stub)
- Hook: when execution paused, write to `approvals` table; when approved, resume graph
- Update `ui/src/pages/Inbox/InboxList.tsx` to read from `/approvals` for this chain (feature-flagged by workflow kind)
- Update `ui/src/components/WorkflowTrace.tsx` to render trace from `audit_log` events for v2 workflows
- Side-by-side test: trigger `content_publish` via v1 and v2; both should produce equivalent end state (`social_posts.status='published'`)

**Exit criteria:** `content_publish` runs end-to-end on the new runner; inbox shows approvals correctly; trace renders.

**Sub-plan:** `5.path-b-phase-1-content-publish.md`

### Phase 2 — Migrate `rev_rec_monthly` (~1.5 weeks)

Most complex chain — branching paths, on_approve callback that spawns a fresh workflow, two execution gates.

- Graph nodes: `validate_and_sync`, `propose_configure_incomplete` (interrupt_before, optional), `compute_entries`, `propose_write_entries` (interrupt_before), `write_entries`
- Conditional edges: from `validate_and_sync`, route to `propose_configure_incomplete` if incomplete else `compute_entries`
- The "on approve, requeue" semantics: approval handler calls `spawn_workflow(REV_REC_KIND, ...)` with `parent_workflow_id` linking back. Audit log captures the spawn.
- Migrate the rev_rec test suite to v2 patterns

**Exit criteria:** `rev_rec_monthly` runs both paths (incomplete → checkpoint → requeue, ready → write entries) on v2.

**Sub-plan:** `6.path-b-phase-2-rev-rec.md`

### Phase 3 — Migrate `outreach_chain` and `content_creation` (~1.5 weeks)

Both have critique loops. This is where LangGraph cycles really pay off.

- `outreach_chain` graph: `pull_hubspot → web_search → consolidate → retrieve_kb → draft → voice_critique → accuracy_critique → propose_send (interrupt_before) → gmail_send`
  - Critique nodes have conditional edges back to `draft` on fail with budget; to next node on pass; to a `failed` terminal on budget exhausted
  - Attempt counts in graph state
- `content_creation` graph: `interpret_brief → draft_post → voice_review (cycle back to draft on fail)`
- Migrate both test suites

**Exit criteria:** both chains run end-to-end on v2; critique loops produce the same number of retry-of attempts as before.

**Sub-plan:** `7.path-b-phase-3-outreach-content.md`

### Phase 4 — Multi-agent primitives (~1 week)

Goal: stand up the agent-communication primitives so future graphs can use them. No existing chain needs them yet.

- Migration `0012_create_agent_messages.sql`
- `app/services/agent_messages.py`: `send_message`, `read_thread`, `get_messages_for_workflow`
- Extend `agent_invoke.py` so an agent's tool call can target another agent (e.g., `tools/agent_tools.py` exposes `ask_agent` as a tool)
- Build a sample multi-agent graph (not a real chain — a demo) showing: supervisor agent decides which specialist to call, specialist proposes, supervisor reviews, supervisor approves or rejects
- Tests for the messaging primitives

**Exit criteria:** an agent in one node can send a message to another agent; the receiving agent reads the thread and responds; full message history queryable.

**Sub-plan:** `8.path-b-phase-4-multi-agent.md`

### Phase 5 — Cleanup and decommission (~1 week)

Goal: delete the old code, drop the old tables, finalize docs.

- Delete `app/orchestrator/` (old orchestrator)
- Delete `app/orchestrator_v2/` → rename to `app/orchestrator/`
- Migration `0013_drop_actions_table.sql` — after archival snapshot
- Migration `0014_drop_workflow_pattern_columns.sql` — drop `pattern` and `current_step` from `workflows`
- Drop `StepKind` enum, `step_kind` column references everywhere
- Drop the `actions` related routers/services/UI
- Final docs pass

**Exit criteria:** `actions`, `step_kind`, and `pattern` are entirely gone from the codebase; old orchestrator deleted; full test suite green.

**Sub-plan:** `9.path-b-phase-5-cleanup.md`

## Risk mitigation

**Side-by-side execution during migration.** Old orchestrator and new runner coexist from Phase 0 through Phase 5. We never have a moment where the system can only do one or the other.

**One chain at a time.** Each phase migrates one chain end-to-end before moving on. If Phase 2 reveals a problem with the runner, only one chain is affected; the others stay on v1.

**Tests gate every phase.** A phase doesn't ship until both the new code passes its tests AND the chains we haven't migrated yet still pass their old tests.

**Pre-existing test infrastructure issues.** The conftest fixture is currently broken (`agents.name` column from migration 0006). Fix that as Phase 0 step zero so we have a real safety net during the rewrite.

**Approval inbox migration is the highest-risk UI change.** Mitigation: dual-source the inbox during Phases 1–4. The component reads `actions` for not-yet-migrated chains and `approvals` for migrated ones. Drop the dual source in Phase 5.

**Critique loops are the trickiest framework concept to preserve.** Mitigation: write the LangGraph cycle pattern as a small standalone proof-of-concept in Phase 0 before any chain depends on it.

**LangGraph version drift.** Pin to a specific minor version in `pyproject.toml`. Don't auto-upgrade.

## Open questions to decide before Phase 0

These need answers to shape Phase 0's sub-plan. None of them block writing this master plan, but they need answers before code.

1. **Checkpointer backend.** Use `langgraph-checkpoint-postgres` against the same Supabase instance, or store graph state in a JSONB column on `workflows`? The first is more LangGraph-native; the second keeps state queryable from our existing tooling. **Default recommendation:** use `langgraph-checkpoint-postgres` — it's the path of least resistance and gives us time-travel debugging for free.

2. **`actions` table fate during migration.** Drop entirely in Phase 5, or keep read-only as a historical archive? **Default recommendation:** snapshot to a `actions_archive` table in Phase 5, then drop the live table.

3. **Trigger surface.** n8n currently calls FastAPI endpoints to trigger workflows. Does that endpoint shape change? **Default recommendation:** keep `POST /workflows/{kind}` as the public API; internally it dispatches to v1 or v2 runner based on which kind is registered.

4. **Approval routing.** Should `approvals` carry an `assigned_to` field for future multi-user routing, or stay single-user for now? **Default recommendation:** add the column nullable now; ignore it in v1.

5. **Audit event taxonomy.** What events does v2 emit? Need a documented list before Phase 0 implements the hooks. **Default recommendation:** start with `workflow.{started,completed,failed,paused,resumed}`, `node.{entered,exited,failed}`, `approval.{requested,granted,rejected,executed}`, `agent.{invoked,completed}`, `subworkflow.{spawned,completed}`. Refine as needed.

## Verification at each phase

Every phase ends with the same three checks:

1. **Tests:** full `pytest` green; new tests for new primitives; old tests preserved for un-migrated chains.
2. **End-to-end:** trigger every chain (migrated and not-yet-migrated) against local Supabase; verify identical end-state to v1.
3. **UI:** open the approval inbox, confirm pending items render correctly with payload edits + reject reasons working.

A phase doesn't merge unless all three pass.
