# Phase 0 — Foundations

⚠️ **Medium** — One week of focused work. Foundational, but isolated: nothing in `app/orchestrator/` or the `actions` flow is touched. The new code lives next to the old code in `app/orchestrator_v2/` and the new `approvals` table coexists with `actions`.

> **Parent plan:** [`3.langgraph-multi-agent-rearchitecture.md`](./3.langgraph-multi-agent-rearchitecture.md). Read that first for the why.

## Goal

Stand up the new infrastructure (LangGraph runtime, `approvals` table, agent-invocation primitive, sub-workflow primitive) so Phase 1 can migrate `content_publish` without doing infrastructure work and chain-migration work in the same sub-plan.

**Exit criteria:**
- `pytest` is fully green, including the previously-broken conftest fixture.
- `langgraph` and `langgraph-checkpoint-postgres` are installed and pinned.
- New tables (`approvals`) and column (`workflows.parent_workflow_id`) exist locally and in `docs/SCHEMA.md`.
- `app/orchestrator_v2/runner.py` can register a trivial `StateGraph`, run it to completion, and emit the documented audit events.
- `app/orchestrator_v2/runner.py` can pause on `interrupt_before`, write a row to `approvals`, accept an approval via the new endpoint, and resume.
- `app/orchestrator_v2/agent_invoke.py` can invoke any registered agent from inside a node.
- `app/orchestrator_v2/spawn.py` can spawn a child workflow with `parent_workflow_id` linkage.
- A small standalone proof-of-concept LangGraph cycle (the critique-loop pattern) exists with a passing test, so Phase 3 isn't designing the cycle pattern from scratch.
- `PRD.md`, `CLAUDE.md`, `docs/SCHEMA.md`, and `PROGRESS.md` reflect the changes (per master plan's doc-update matrix).
- Old orchestrator and `actions` flow are completely untouched. v1 chains continue to run.

## Step 5: The v2 runner

### `app/orchestrator_v2/state.py`

Define the conventions for graph state TypedDicts. Recommendation: every graph's state inherits a small base shape:

```python
from typing import TypedDict, NotRequired

class BaseGraphState(TypedDict):
    workflow_id: NotRequired[str]   # injected by the runner before first node
    parent_workflow_id: NotRequired[str | None]
    # Graphs add their own keys. Avoid putting LangGraph-internal stuff here.
```

State is the **graph's** runtime data (drafts, retrieved KB chunks, attempt counts). Long-lived state goes to its own table (e.g., `social_posts`), not into graph state.

### `app/orchestrator_v2/runner.py`

The runner is the single entry point for v2 work. Public surface:

```python
class V2Runner:
    """Owns the LangGraph compiled-graph registry and the checkpointer."""

    def register(self, kind: str, graph_factory: Callable[[], StateGraph]) -> None: ...
    def is_registered(self, kind: str) -> bool: ...

    async def start(self, kind: str, *, initial_state: dict, workflow_id: UUID, ...) -> None:
        """Persist a workflows row, then drive the graph until first interrupt or completion."""

    async def resume(self, workflow_id: UUID) -> None:
        """Resume a paused graph (after approval) until next interrupt or completion."""

    async def shutdown(self) -> None: ...
```

Internals:
- Wraps each registered `StateGraph` in `graph.compile(checkpointer=..., interrupt_before=[...])`. The list of interrupt nodes is declared by the graph factory.
- On entry to every node, emits `NODE_ENTERED`; on successful exit, `NODE_EXITED`; on exception, `NODE_FAILED` and re-raises (LangGraph captures it; the runner translates to `WORKFLOW_FAILED`).
- When a node `interrupt_before` is hit, the runner inspects the node's pending state to extract the proposed approval payload, calls `approvals.create_pending`, and emits `WORKFLOW_PAUSED` + `APPROVAL_REQUESTED`. Convention: each interrupted node returns/yields a dict with `_propose: { action_type, proposed_payload, summary, reasoning, risk_level, agent_slug }`. The runner reads that key, never inspects business state.
- `resume(workflow_id)`: looks up the matching approval, applies any `executed_payload` overrides, calls `graph.ainvoke(None, config={..., "configurable": {"thread_id": workflow_id}})` to continue. On normal completion, emits `WORKFLOW_COMPLETED`; calls `mark_executed` on the approval that gated the now-completed segment.
- All audit events go through `app/services/audit.write_audit_event()` — no new audit infrastructure.
- Initialized at app startup in `app/main.py` lifespan; `register()` is called for every registered v2 graph (none in Phase 0; the runner just exists with an empty registry plus the critique POC).

Trigger dispatch from existing endpoints (master-plan resolution #3):
- `POST /workflows/outreach` and `POST /workflows`: check `runner.is_registered(kind)` first; if true, call `runner.start(...)`; else fall back to the v1 orchestrator. Phase 0: nothing real is registered, so behavior is unchanged.

### `app/orchestrator_v2/approval.py`

A small adapter that the runner calls into. Routes between `app/services/approvals.py` (DB) and the runner (resume hooks). Lives in orchestrator_v2 because the resume orchestration is graph-aware.

### `app/orchestrator_v2/agent_invoke.py`

```python
async def invoke_agent(slug: str, input: dict, ctx: NodeContext) -> dict:
    """Uniform entry point for agent invocation from a graph node, sub-agent, or chat surface.

    - Looks up the agent by slug in the DB.
    - Constructs the LLM call with the agent's system_prompt and allowed_tools.
    - Emits AGENT_INVOKED, runs, emits AGENT_COMPLETED / AGENT_FAILED.
    - Returns the agent's structured response.
    """
```

`NodeContext` carries `workflow_id`, `parent_workflow_id`, `pool`, and a logger. Keep it small. Phase 4 extends this to support agent-to-agent messaging via `agent_messages`.

### `app/orchestrator_v2/spawn.py`

```python
async def spawn_workflow(kind: str, input: dict, *, parent_workflow_id: UUID, initiated_by: str = "system") -> UUID:
    """Spawn a child workflow, link via parent_workflow_id, emit SUBWORKFLOW_SPAWNED.

    Returns the child workflow_id immediately. The child runs to its first interrupt
    or completion in the same task (caller awaits)."""
```

Used by Phase 2's "on approve, requeue" semantics for `rev_rec_monthly`. Built in Phase 0 with a passing test so Phase 2 doesn't have to design it.

## Step 6: Critique-loop proof-of-concept

The cycle pattern is the single highest-risk thing about Phase 3. Building a tiny standalone version now de-risks two phases.

### `app/orchestrator_v2/graphs/_critique_poc.py`

Build a 3-node graph: `draft → critique → publish`. The `critique` node returns either `pass` or `fail`. A conditional edge routes back to `draft` on `fail` while `state["attempts"] < max_attempts`; to `publish` on `pass`; to a `failed` terminal when budget is exhausted. No LLMs — `draft` returns a string from a list, `critique` toggles fail/pass based on attempts.

### `tests/test_critique_loop_poc.py`

Three tests:
1. Pass on first attempt → exits via `publish`, `attempts == 1`.
2. Fail twice, pass on third → exits via `publish`, `attempts == 3`.
3. Fail past budget → exits via `failed` terminal.

This file gets deleted in Phase 5 — it's only here as a pattern reference.
