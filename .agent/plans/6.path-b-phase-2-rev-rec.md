# Phase 2 — Migrate `rev_rec_monthly` to LangGraph (v2 runner)

🔴 **Complex** — most intricate chain in the codebase: branching paths, two interrupt gates, an iteration loop driven by human approval, and 5 external integrations called inline. v2 infrastructure (Phase 0) and the v2 migration pattern (Phase 1, `content_publish`) are both in place, so this is a structurally larger but architecturally familiar phase.

Sub-plan for the master plan at `.agent/plans/3.langgraph-multi-agent-rearchitecture.md` (Phase 2, lines 144–155).

## Context

v1 `rev_rec_monthly` is a 4-step `supervised_automation` chain. It does the right things — sync Harvest→Airtable, validate that every project has the fields we need, surface incomplete projects to a human for triage, recompute revenue, propose write, write — but the implementation has accumulated three problems Phase 2 should resolve:

1. **A real bug.** `_propose_write` looks up its upstream result with `latest_for_step(2)`, which works only when the checkpoint step actually ran. When the checkpoint is skipped (validation passed on first try), step indices shift and the lookup misses. Currently xfailed in `tests/test_rev_rec_chain.py`. Fixed naturally in v2 because LangGraph state is shared across nodes — no sequence-based lookups.
2. **The "iterate until clean" semantics are awkward in v1.** The checkpoint's `on_approve` callback creates a *new* workflow row and resumes it, leaving an audit trail split across multiple workflow_ids linked only by initiated_by string. The user observes one conceptual job (close out the month) but the system records it as N separate workflows.
3. **The chain is hard to read.** Branching via `skip_if` predicates that share state through sequence-indexed lookups is more complicated than it needs to be. v2 expresses the same logic as a graph with one conditional edge and one loop edge.

After Phase 2, `rev_rec_monthly` runs on LangGraph end-to-end. One workflow_id covers the full iteration cycle: validate → (incomplete? → configure checkpoint → loop back to validate; ready? → compute → write checkpoint → write). The bug is gone. The iteration model matches the user's mental model.

## Decisions (locked before drafting)

1. **Iteration pattern: loop edge, not sub-workflow spawn.** One workflow_id covers all configure→validate iterations. Aligns with the propose+execute future where the configure approval will eventually carry actual change-payloads (not just a "re-check" signal). The spawn primitive remains tested in Phase 0 and is available for genuine fan-out scenarios in future work.
2. **v1 coexistence: unregister immediately.** Same approach as Phase 1 — drop `REV_REC_CHAIN` from the v1 registry, leave the chain object defined for Phase 5 deletion, flip the agent tool's dispatch site to `runner.start("rev_rec_monthly", ...)`.
3. **Tests: fresh v2 tests, let v1 rot.** New `tests/test_v2_rev_rec_graph.py`. v1 `tests/test_rev_rec_chain.py` (1 passing + 2 xfailed) gets parametrized out / scheduled for Phase 5 deletion.

## Graph topology

```
                          ┌─ status="ready" ──────────────────────────────────────┐
                          │                                                        │
[entry] → validate_and_sync                                                        ▼
                          │                                                  compute_entries
                          │                                                        │
                          └─ status="incomplete" ─→ propose_configure              │
                                                          │                         │
                                                  [interrupt_before                 │
                                                   apply_configure_or_loop]         │
                                                          │                         │
                                                          ▼ (after approve)         │
                                                  apply_configure_or_loop ─────────┐│
                                                          │                         ││
                                                          └─→ validate_and_sync ◄───┘│
                                                                                     │
                                                                                     ▼
                                                                        propose_write_entries
                                                                                     │
                                                                              [interrupt_before
                                                                               write_entries]
                                                                                     │
                                                                                     ▼ (after approve)
                                                                              write_entries
                                                                                     │
                                                                                     ▼
                                                                                    END
```

`interrupt_before = ("apply_configure_or_loop", "write_entries")`

Two interrupt gates. Conditional edge after `validate_and_sync` based on `state.validation_status`. Loop edge from `apply_configure_or_loop` back to `validate_and_sync`. v1's `skip_if` predicates are replaced by the conditional edge — there's no concept of "skip" because nodes that aren't reached simply don't run.

## State design

```python
class RevRecState(BaseGraphState, total=False):
    # From initial_state / trigger
    date_recognized: NotRequired[str]
    context: NotRequired[dict[str, Any]]

    # Set by validate_and_sync (overwritten on each iteration)
    validation_status: NotRequired[Literal["ready", "incomplete"]]
    incomplete_projects: NotRequired[list[dict]]
    projects: NotRequired[list[dict]]   # full Airtable rows when ready
    month_label: NotRequired[str]

    # Set by compute_entries
    entries: NotRequired[list[dict]]
    total_recognized: NotRequired[float]

    # Pushed in by the runner on approval resume
    executed_payload: NotRequired[dict[str, Any]]

    # Final
    result: NotRequired[dict[str, Any]]
```

LangGraph reducers default to "replace." That's correct here — each iteration's `validate_and_sync` overwrites `validation_status`, `incomplete_projects`, `projects` with fresh values. No accumulation.

## Implementation steps

### Step 1 — Define `app/orchestrator_v2/graphs/rev_rec.py`

Six nodes mirroring v1's logic, restated as state-in/state-out functions. Concrete shapes (abbreviated; see the v1 chain for full bodies):

- **`validate_and_sync(state)`** — reads `state.date_recognized` (default: last day of prev month), runs the duplicate guard (`airtable.get_most_recent_revenue_entry`; raise `ValueError` if a record already exists for the same date), runs `airtable_sync.run_sync(settings)`, fetches projects, classifies each as ready/incomplete. Returns `{validation_status, incomplete_projects, projects, month_label}` and clears stale `incomplete_projects` from prior iterations.

- **`route_after_validate(state) -> str`** — returns `"propose_configure"` if `validation_status == "incomplete"`, else `"compute_entries"`. Used by `add_conditional_edges`.

- **`propose_configure(state)`** — returns `{"_propose": ProposeApproval(action_type="configure_rev_rec_projects", agent_slug="revenue-recognition", risk_level="low", summary=..., proposed_payload={"date_recognized": ..., "incomplete_projects": [...], "context": ...})}`. Payload shape **must match v1** so the existing inbox renderer at `ui/src/pages/Inbox/InboxList.tsx:74-90` works unchanged.

- **`apply_configure_or_loop(state)`** — POC behavior: no-op. The human's approval signaled "I fixed the data externally; please re-validate." After this node runs, the graph routes back to `validate_and_sync` via a deterministic edge.

  > **Designed for upgrade.** When propose+execute lands, this node will read `state["executed_payload"]` (the human-edited update list) and apply each update to Airtable before looping. The graph topology stays identical; only this node body grows. Document the upgrade contract in the docstring.

- **`compute_entries(state)`** — reads `state.projects` and `state.date_recognized`, calls `forecast.get_scheduled_hours_by_harvest_id`, `harvest.get_invoice_totals_by_project`, and `harvest.get_time_entries` per project, calls `app.services.revenue.calc_revenue` to derive revenue per project, returns `{entries, total_recognized}`. Replaces v1's `latest_for_step(0)` lookup with direct state read.

- **`propose_write_entries(state)`** — returns `{"_propose": ProposeApproval(action_type="write_rev_rec", risk_level="low", proposed_payload={"date_recognized": state.date_recognized, "total_recognized": state.total_recognized, "entries": state.entries})}`. **Same payload shape as v1** for inbox UI compatibility (`InboxList.tsx:93-137`). This is where v1's bug existed; v2 reads from state directly so there's nothing to misalign.

- **`write_entries(state)`** — reads `state.executed_payload.entries` (or falls back to `state.entries` if no edit happened — though by convention executed_payload is always populated post-approve; runner pushes proposed_payload through if user didn't edit), strips `_`-prefixed internal fields, calls `airtable.create_revenue_records(settings, clean_entries)`. Returns `{result: {records_created, airtable_ids}}`.

Wire it up:

```python
g.add_node("validate_and_sync", validate_and_sync)
g.add_node("propose_configure", propose_configure)
g.add_node("apply_configure_or_loop", apply_configure_or_loop)
g.add_node("compute_entries", compute_entries)
g.add_node("propose_write_entries", propose_write_entries)
g.add_node("write_entries", write_entries)

g.set_entry_point("validate_and_sync")
g.add_conditional_edges(
    "validate_and_sync",
    route_after_validate,
    {"propose_configure": "propose_configure", "compute_entries": "compute_entries"},
)
g.add_edge("propose_configure", "apply_configure_or_loop")
g.add_edge("apply_configure_or_loop", "validate_and_sync")  # loop back
g.add_edge("compute_entries", "propose_write_entries")
g.add_edge("propose_write_entries", "write_entries")
g.add_edge("write_entries", END)

return GraphSpec(graph=g, interrupt_before=("apply_configure_or_loop", "write_entries"))
```

DB and integration access inside nodes: acquire from `app.db.get_pool()` for the audit DB; call integrations directly per existing chain conventions. No `NodeContext` needed (no agent invocations in this graph).

## Risks & open watches

- **Infinite loop (configure → approve → still incomplete → configure → approve → still incomplete → ...)**. v1 had this same risk; humans bound the loop in practice. No framework-level guard. Document in the node docstring; revisit if it becomes a real issue.
- **Stateful airtable mocks across iterations.** The "iterate" test needs to mutate the integration mock between iterations. Make sure the test fixture supports this (mutable list / state object), not stateless monkeypatches.
- **AsyncPostgresSaver checkpoint accumulation.** Each loop iteration writes a new checkpoint snapshot to Postgres. After many iterations the workflow's checkpoint history grows. Acceptable for POC; production-scale rev_rec rarely loops more than 2–3 times.
- **Phase 1's `test_v2_content_publish_graph.py` is still unverified.** Don't let Phase 2 testing get blocked behind it. If the same conftest interaction blocks Phase 2 tests, escalate and treat both at once.
- **The "approve = re-check signal" idiom is officially codified here.** Anyone reading the graph for the first time will wonder why `apply_configure_or_loop` is a no-op. The docstring on that node is load-bearing — it must explain the POC vs. propose+execute distinction explicitly.
