# 2. Month-End Invoice Agent — MVP

⚠️ **Medium** — Single-pass executable but multi-subsystem (backend chain + service + integration + migration + frontend preview + tests + docs). All changes follow existing patterns. Natural break point if iteration is desired: do **§A. Backend** first as one PR, then **§B. Frontend + docs** as a follow-up.

---

## Context

The PRD currently lists `❌ Invoice operations (retired; re-implement when use case lands)`. The use case has now landed: `month_end_invoice_agent_design_pattern.md` specifies a Harvest-driven month-end billing assistant. The manual process today is a multi-system reconciliation — review prior Harvest invoices, look at active projects, cross-reference a hosting spreadsheet, and create draft invoices in Harvest by hand.

**MVP scope (locked with user):**
- **T&M projects only.** Hosting / fixed-fee / retainer flagged in coverage warnings as "skipped — not in MVP scope".
- **Hosting spreadsheet deferred** to phase 2 (no spreadsheet integration exists today; the design doc treats it as source-of-truth, but adding gspread/openpyxl/Graph is a separable build).
- **One batched approval.** Single checkpoint surfaces all proposed invoice groups; human edits via `EditBodyModal`; one execution batch-creates Harvest drafts. Mirrors `rev_rec_monthly`.
- **No memory/rules learning, no one-off mode, no scheduled runs.** Phase 2.

**Intended outcome.** A user runs the chain (chat tool or `POST /workflows`), reviews a packet of T&M invoice groups + coverage warnings + duplicates + skipped (non-T&M) projects, edits if needed, approves once, and the system creates Harvest draft invoices. **Drafts only — never sends.**

**Why the design is clean.** Harvest's `POST /v2/invoices` supports `line_items_import.time`, which auto-generates line items from uninvoiced time entries for a given `(client_id, project_ids, from, to)` tuple. We don't compute line items ourselves — we identify which projects belong on which invoice and let Harvest do the math. This eliminates a large category of computation bugs from MVP scope and is the reason this fits cleanly in a single plan.

## Approach

### Chain shape (mirrors `rev_rec_monthly`)

`Chain(kind="month_end_invoice", pattern=supervised_automation, agent_slug="month-end-invoicing", steps=...)` with three steps:

| # | Name | step_kind | Behavior |
|---|---|---|---|
| 0 | `_resolve_and_propose` | `tool_call` | Resolves target period, pulls Harvest data, builds T&M invoice groups, runs coverage + duplicate checks, returns review packet. Auto-progresses. |
| 1 | `_propose_review_packet` | `checkpoint` | Surfaces the review packet for human approval; editable via `EditBodyModal`. `action_type="propose_month_end_invoices"`. |
| 2 | `_create_drafts` | `execution` | On approval, calls `harvest.create_invoice()` once per approved group in `executed_payload`. `action_type="create_harvest_draft_invoices"`. |

No `skip_if` branches needed — even with zero proposals, the packet still shows coverage/skipped, and an empty-approval execution is a clean no-op. (Compare rev-rec, which branches on incomplete projects; we have no equivalent "fix this externally and re-run" loop.)

### Step 0 logic (`_resolve_and_propose`)

1. **Target period** from `workflow.trigger_payload["target_month"]` (e.g. `"2026-04"`); default = previous calendar month.
2. **Comparison period** = calendar month before target.
3. **Pull Harvest data** (mostly **new methods** — see §A.3):
   - `harvest.get_active_projects(cfg)` — existing
   - `harvest.list_invoices(cfg, from=target_start, to=target_end)` — new
   - `harvest.list_invoices(cfg, from=comparison_start, to=comparison_end)` — new
   - `harvest.list_time_entries(cfg, from=target_start, to=target_end)` — new (returns raw entries, not summed)
4. **Filter to T&M.** Inspect each project's `bill_by` and `budget_by` from Harvest. Non-T&M projects → `skipped` array. Service should be defensive about field names — Harvest exposes `bill_by` values like `"People"`, `"Project"`, `"Tasks"`, `"none"`; T&M typically `"People"` or `"Tasks"`.
5. **Per-project evidence** — active T&M flag, target-month hours, has prior comparison-month invoice, prior invoice's project grouping.
6. **Build invoice groups deterministically:**
   - Group by `(client_id, prior_invoice_grouping)`. If two T&M projects for the same client appeared on the same comparison-month invoice, group them again.
   - Project with billable hours and no prior invoice → standalone group, `confidence: "medium"`.
   - Project with prior invoice but no current hours → not proposed; coverage warning, `risk: "high"`.
7. **Duplicate detection** (per design doc §5.9). Idempotency key = `(client_id, sorted(project_ids), target_period, "T&M")`. If an existing target-month invoice already covers a `(client, projects)` tuple, mark it `already_invoiced` instead of `proposed`.
8. **Coverage check.** Every active T&M project lands in exactly one of: `proposed`, `already_invoiced`, `no_hours_no_prior` (low risk), `no_hours_had_prior` (high risk).
9. **Reason templates** (deterministic, no LLM in MVP):
   - High: `"Same client and grouping as {comparison_month} invoice #{id}; {n_hours} billable hours in {target_month}."`
   - Medium: `"New T&M project this month — no prior invoice pattern; {n_hours} hours."`
10. **Return** the review packet (shape in §C).

### Step 2 logic (`_create_drafts`)

For each `invoice_groups[i]` in `ctx.executed_payload`:

```python
await harvest.create_invoice(cfg, {
    "client_id": group["client_id"],
    "subject": f"{group['client_name']} — {group['period_label']} Professional Services",
    "issue_date": target_end_iso,
    "line_items_import": {
        "time": {
            "summary_type": "project",
            "project_ids": group["project_ids"],
            "from": target_start_iso,
            "to": target_end_iso,
        }
    },
})
```

Collect `(invoice_id, draft_url)` per group; return summary in `result`. **Drafts only** — never call `POST /v2/invoices/{id}/messages` (which sends).

**File:** `app/services/invoicing.py` (new)

Pure functions. No DB writes. Inputs are Harvest dicts; outputs are the review-packet structures. Step 0 in the chain is a thin shim that pulls data and delegates here. Keeps logic unit-testable in isolation.

Suggested public surface:
```python

def detect_duplicates(groups: list[InvoiceGroup], existing_target_invoices: list[dict]) -> tuple[list[InvoiceGroup], list[AlreadyInvoiced]]
def compute_coverage(active_projects, groups, already_invoiced) -> list[CoverageWarning]
def assemble_packet(...) -> ReviewPacket
```

Use Pydantic v2 models for the structures (no `class Config` — `model_config`).

**Validation:** New `tests/test_invoicing_service.py` covers:
- T&M classification of representative `bill_by` values (`"People"`, `"Tasks"`, `"none"`, `"Project"`).
- Grouping when two projects shared a prior invoice → grouped this month.

## §C. Payload shape — the review packet

`proposed_payload` for the checkpoint action; same shape the inbox renders and the human edits:

```json
{
  "target_period": {"label": "April 2026", "from": "2026-04-01", "to": "2026-04-30"},
  "comparison_period": {"label": "March 2026", "from": "2026-03-01", "to": "2026-03-31"},
  "scope_note": "MVP: T&M projects only. Hosting, retainer, and fixed-fee projects are listed in skipped[] and must be invoiced manually.",
  "invoice_groups": [
    {
      "group_id": "acme-2026-04-tm",
      "client_id": 123,
      "client_name": "Acme Co.",
      "invoice_type": "T&M",
      "project_ids": [123, 456],
      "project_names": ["Website Redesign", "SEO Support"],
      "estimated_hours": 80.0,
      "estimated_amount": 12000.0,
      "reason": "Same client and grouping as March invoice #4521; 80 billable hours in April.",
      "confidence": "high",
      "recommended_action": "approve"
    }
  ],
  "already_invoiced": [
    {"client_name": "Beta LLC", "invoice_id": 9876, "issue_date": "2026-04-15", "amount": 5000.0,
     "reason": "Existing April T&M invoice covers all expected projects for this client."}
  ],
  "coverage_warnings": [
    {"project_name": "Acme Strategy", "client_name": "Acme Co.", "reason": "Active T&M project, no April hours", "risk": "low"},
    {"project_name": "Gamma Platform", "client_name": "Gamma Inc.", "reason": "Had March invoice, no April hours", "risk": "high"}
  ],
  "skipped": [
    {"project_name": "Acme Hosting", "reason": "Hosting project — hosting integration deferred to phase 2"},
    {"project_name": "Delta Retainer", "reason": "Retainer project — not in MVP scope"}
  ],
  "duplicates": []
}
```

When the human edits via `EditBodyModal`, they remove/edit `invoice_groups` entries. Step 2 iterates `executed_payload.invoice_groups`. `proposed_payload` stays intact for audit (per `docs/SCHEMA.md:96`).
