# Phase 4 — Multi-agent primitives (agent_messages, ask_agent, demo graph)

⚠️ **Medium** — schema migration, one new service, one new tool, one demo graph, one ToolContext extension. No production-chain disruption (Phases 1–3 already migrated everything). Focused, additive work.

Sub-plan for the master plan at `.agent/plans/3.langgraph-multi-agent-rearchitecture.md` (Phase 4, lines 175–185).

## Context

After Phase 3, every v1 chain runs on LangGraph. Workflows now have rich primitives:

- `runner.start / resume` — graph lifecycle
- `invoke_agent(slug, ...)` — single-turn agent invocation with audit events
- `spawn_workflow(kind, ..., parent_workflow_id=...)` — sub-workflow linkage
- Critique loops, conditional edges, interrupts, all proven across four kinds

**What's missing:** an agent in one node can't *talk to* another agent through a queryable channel. `invoke_agent` returns text but doesn't record a turn-by-turn conversation. There's no thread, no "see what was asked", no "supervisor delegates to specialist" pattern with traceable history.

Phase 4 adds the messaging substrate that makes future multi-agent work first-class:

1. **`agent_messages` table** — every agent-to-agent turn lives in one place, threadable, queryable per workflow.
2. **`ask_agent` tool** — a tool any agent can opt into (via `allowed_tools`) to delegate a question to another agent. Records both the outgoing prompt and the incoming reply as message rows.
3. **`workflow_id` on `ToolContext`** — so node-driven tool calls can link their messages to the workflow audit trail.
4. **Demo graph** — a `_multi_agent_demo` graph (alongside `_critique_poc`) that exercises `agent_messages` + `ask_agent` end-to-end with a supervisor → specialist → supervisor pattern. Registered like the critique POC; deleted whenever it's no longer useful.

Exit criteria from the master plan: "an agent in one node can send a message to another agent; the receiving agent reads the thread and responds; full message history queryable." Phase 4 hits all three.

## Decisions (locked before drafting)

1. **Threads are correlation keys, not full conversations.** A `thread_id` ties messages together but each "turn" of agent-to-agent talk is single-shot at the LLM layer (we use `invoke_agent` which is single-turn). Phase 4 *records* a thread but doesn't *use* prior turns to seed the next agent's prompt — the receiver sees only the current question. Multi-turn context-passing is a Phase 5+ refinement that doesn't change the schema.
2. **`ask_agent` tool wraps `invoke_agent`, not Anthropic native tool-use.** The supervisor pattern in the demo graph runs as: node parses LLM response as JSON → node calls `invoke_agent` (specialist) → node records messages. We do **not** extend `invoke_agent` to handle Anthropic native tool-use loops in this phase. That's a heavier change with its own design space (token accounting, max-iterations safety, partial-result handling) and isn't on the critical path for the exit criteria.
3. **Demo graph reuses existing agents.** No new agent classes added to `AGENTS`. The demo wires existing agent slugs (e.g., `outreach-agent` as a specialist, `voice-critic` as another) so the ergonomic surface (slug → existing system prompt) stays consistent. The demo is illustrative; its agent choices are arbitrary.
4. **No new agent added to `allowed_tools`.** `ask_agent` is registered in the tool registry but no existing agent enables it in this phase. Wiring it into a chat agent (e.g., `content-orchestrator`) is a follow-up if and when chat actually wants delegation. Keeping Phase 4 surface-area minimal.
5. **Tests are NOT deferred.** Unlike Phase 1–3 graph tests (blocked by AsyncPostgresSaver/conftest interaction), Phase 4's service-layer tests use only the asyncpg pool — no LangGraph checkpointer, no provider conflict. Should run in the existing harness without diagnostic work.
6. **Migration number 0013.** Master plan says 0012, but 0012 was already used in Phase 1 as the LangGraph checkpoint-tables marker. This Phase 4 migration becomes 0013. Phase 5's two cleanup migrations renumber accordingly to 0014 and 0015.

## Schema — `agent_messages`

```sql
create table agent_messages (
    id                uuid primary key default gen_random_uuid(),
    thread_id         uuid not null,
    workflow_id       uuid null references workflows(id) on delete cascade,
    from_agent_slug   text not null,
    to_agent_slug     text not null,
    content           text not null,
    metadata          jsonb not null default '{}'::jsonb,
    created_at        timestamptz not null default now()
);

create index agent_messages_thread_idx on agent_messages(thread_id, created_at);
create index agent_messages_workflow_idx on agent_messages(workflow_id) where workflow_id is not null;
create index agent_messages_to_idx on agent_messages(to_agent_slug);
```

- `thread_id` groups messages exchanged within one delegation. The sender generates a fresh UUID for the first message of a new conversation; subsequent turns reuse it.
- `workflow_id` is nullable so chat-driven `ask_agent` calls (no workflow context) still record cleanly. When set, ON DELETE CASCADE keeps the table clean during workflow purges.
- Indexes: thread reads are the hot path; the `where workflow_id is not null` partial index keeps the per-workflow lookup tight.
- Roles aren't represented (no `role` column). This isn't chat — it's agent-to-agent. The pair `from_agent_slug` / `to_agent_slug` is enough to reconstruct turn ordering within a thread by `created_at`.

## Service — `app/services/agent_messages.py`

```python
async def send_message(
    pool_or_conn,
    *,
    from_agent_slug: str,
    to_agent_slug: str,
    content: str,
    thread_id: UUID | None = None,
    workflow_id: UUID | None = None,
    metadata: dict[str, Any] | None = None,
) -> dict[str, Any]:
    """Insert a row. If thread_id is None, generate a fresh one. Returns the row dict."""

async def read_thread(pool, thread_id: UUID) -> list[dict[str, Any]]:
    """All messages in this thread, oldest first."""

async def get_messages_for_workflow(pool, workflow_id: UUID) -> list[dict[str, Any]]:
    """All messages tied to a workflow, oldest first. Returns [] if none."""
```

`send_message` accepts either a pool or an existing connection — same dual API pattern as `social_posts`. Audit events are NOT written for individual messages (the volume would dominate audit_log); instead, the demo graph emits a `node.exited` event per node as usual. If we later want a "supervisor delegated to specialist" event, that's a constant in `events.py` plus a one-line `write_audit_event` call from the demo node — out of scope for Phase 4 base.

## Tool — `app/tools/agent_tools.py`

```python
ASK_AGENT = ToolDefinition(
    name="ask_agent",
    description=(
        "Ask another agent a question. Returns the agent's reply as text plus the "
        "thread_id used to record both messages. Pass an existing thread_id to "
        "continue a conversation; omit it to start a new thread."
    ),
    input_schema={
        "type": "object",
        "properties": {
            "target_slug": {"type": "string", "description": "Slug of the agent to ask."},
            "prompt": {"type": "string", "description": "The question or request."},
            "thread_id": {"type": "string", "description": "Optional thread UUID to continue a prior exchange."},
        },
        "required": ["target_slug", "prompt"],
    },
    execute=_ask_agent,
)

async def _ask_agent(ctx: ToolContext, *, target_slug: str, prompt: str, thread_id: str | None = None, **_):
    pool = await get_pool()
    thread_uuid = UUID(thread_id) if thread_id else uuid4()
    workflow_id = ctx.workflow_id  # None when invoked from chat

    # 1. Record the outgoing question.
    await agent_messages.send_message(
        pool,
        from_agent_slug=ctx.agent_slug,
        to_agent_slug=target_slug,
        content=prompt,
        thread_id=thread_uuid,
        workflow_id=workflow_id,
    )

    # 2. Get the answer (single-turn).
    response = await invoke_agent(
        target_slug,
        {"prompt": prompt, "max_tokens": 800},
        NodeContext(workflow_id=workflow_id) if workflow_id else None,
    )
    answer = response["text"]

    # 3. Record the answer.
    await agent_messages.send_message(
        pool,
        from_agent_slug=target_slug,
        to_agent_slug=ctx.agent_slug,
        content=answer,
        thread_id=thread_uuid,
        workflow_id=workflow_id,
    )

    return {"answer": answer, "thread_id": str(thread_uuid)}
```

The existing `_ALL_TOOLS` list in `app/tools/__init__.py` gets `ASK_AGENT` appended.

## Demo graph — `app/orchestrator_v2/graphs/_multi_agent_demo.py`

Mirrors `_critique_poc.py` in style and intent: standalone, illustrative, registered alongside production graphs but underscore-prefixed for clarity. **No interrupts** — runs end-to-end on a single `runner.start` call.

Topology:

```
[entry] → supervisor_decide → specialist_propose → supervisor_review
                                                          │
                                                  (route_after_review)
                                                          │
                                                ┌─────────┼─────────┐
                                              approve              reject
                                                │                    │
                                                ▼                    ▼
                                              END                  END
```

State:

```python
class MultiAgentDemoState(BaseGraphState, total=False):
    workflow_id: NotRequired[str]
    user_question: NotRequired[str]
    chosen_specialist_slug: NotRequired[str]
    proposal: NotRequired[str]
    review: NotRequired[dict[str, Any]]   # {decision: "approve"|"reject", reasoning: str}
    thread_id: NotRequired[str]            # set on first send_message; reused thereafter
    result: NotRequired[dict[str, Any]]
```

Nodes (concise):

- **`supervisor_decide(state)`** — call `invoke_agent("outreach-agent", {"prompt": "<chooser>"})` (acts as supervisor for the demo). Parse a JSON response shape `{specialist_slug: "voice-critic"|"accuracy-critic", question: "..."}`. Record both outgoing & incoming messages via `agent_messages.send_message` (writes 2 rows; thread_id generated here). Return `{chosen_specialist_slug, proposal_question, thread_id}`.
- **`specialist_propose(state)`** — call `invoke_agent(state["chosen_specialist_slug"], {"prompt": state["proposal_question"]})`. Record the outgoing question + incoming proposal as 2 more rows in the same thread. Return `{proposal: text}`.
- **`supervisor_review(state)`** — call `invoke_agent("outreach-agent", {"prompt": "Review this proposal: ... Reply JSON {decision: 'approve'|'reject', reasoning: '...'}"})`. Record 2 more messages. Return `{review: {...}}`.
- **`route_after_review(state) -> str`** — `state["review"]["decision"] == "approve"` → END, else → END (both terminate; the routing exists to demonstrate the conditional edge pattern). Could split into `accept` / `reject` sink nodes if we want distinct audit events, but for a demo END is fine.

Wire it up:

```python
g.set_entry_point("supervisor_decide")
g.add_edge("supervisor_decide", "specialist_propose")
g.add_edge("specialist_propose", "supervisor_review")
g.add_conditional_edges("supervisor_review", route_after_review, {END: END})
return GraphSpec(graph=g)  # no interrupts
```

By workflow end, `agent_messages.get_messages_for_workflow(workflow_id)` returns ~6 rows showing the full delegation arc:

1. `outreach-agent → voice-critic` (or whichever specialist): "Here's the question, what do you propose?"
2. `voice-critic → outreach-agent`: "I propose ..."
3. `outreach-agent → voice-critic`: "Now propose given context X"  *(from specialist_propose's outbound)*
4. `voice-critic → outreach-agent`: "Proposal text"
5. `outreach-agent → outreach-agent`: "Review this proposal"  *(self-talk for supervisor_review — fine; the demo isn't pretending to be human-quality dialog)*
6. `outreach-agent → outreach-agent`: review JSON response

The exact count depends on exactly how each node frames its turn. The demo's value is that the rows exist, are queryable, and demonstrate the pattern.

## Risks & open watches

- **Demo's reliance on real Anthropic calls.** The demo invokes `outreach-agent` (Anthropic) as supervisor and at least one critic agent. If `ANTHROPIC_API_KEY` isn't set, the demo errors. Acceptable — we're not committing the demo to CI; it's a manual smoke. The service tests work without LLM access.
- **JSON-parsing fragility for supervisor decisions.** The supervisor agent's prompt asks for JSON; LLMs sometimes return prose. Use `parse_json` (which already handles fenced code blocks and falls back to `{}`); fall back to a default specialist if parsing fails. Don't hard-fail the demo on a JSON edge case.
- **`workflow_id` on ToolContext used only by `ask_agent`.** Risk: dead field for chat path. Acceptable. Phase 5+ will plug more places into it.
- **Audit-log volume.** `ask_agent` does NOT write per-message audit events. The `node.exited` event from the runner is enough granularity for now. If we later want per-turn audit visibility, add `AGENT_MESSAGE_SENT` constant in `events.py` and a one-line `write_audit_event` call in `send_message`.
- **Unbounded conversation length.** A demo node could keep calling `ask_agent` with the same thread_id and the row count would grow indefinitely. The demo doesn't loop. Real graphs that wrap a multi-turn dialog will need their own iteration cap (same pattern as critique-loop max_attempts). Document in `ask_agent` docstring.
- **No new agent classes.** Reusing `outreach-agent` as a "supervisor" is a stretch — that agent's system prompt is about email outreach, not delegation. The supervisor prompt the demo node sends will override the system prompt's intent, but responses may be tinged by the original prompt. Acceptable for a demo. If the demo's behavior is too noisy, swap to a barebones agent class added in `app/agents/_demo.py`. Re-evaluate after first run.
- **Migration number drift.** Master plan called this 0012; we use 0013 because Phase 1 took 0012 as the LangGraph checkpoint marker. Master plan's Phase 5 references migrations 0013 and 0014 (drop_actions, drop_pattern_cols); those become 0014 and 0015. Update Phase 5's plan when it's drafted.
