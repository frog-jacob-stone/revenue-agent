# Phase 3 — Migrate `outreach_chain` and `content_creation` to LangGraph (v2 runner)

🔴 **Complex** — two chains, both with critique-driven cycles. Outreach has the most intricate v2 graph in the migration so far: 9 nodes, two critique loops sharing a single draft node, two independent attempt budgets, plus the existing approval gate. Content creation is the simpler sibling — same loop pattern as the critique POC.

Sub-plan for the master plan at `.agent/plans/3.langgraph-multi-agent-rearchitecture.md` (Phase 3, lines 159–175).

## Context

After Phase 2 the only remaining v1 chains are `outreach_chain` and `content_creation`. Both lean heavily on v1's critique machinery — the part of the bespoke orchestrator hardest to read and reason about. Phase 3 retires that machinery in two graph files modeled on the critique POC (`app/orchestrator_v2/graphs/_critique_poc.py`).

Why now:
- The exit criteria from the master plan say "critique loops produce the same number of retry-of attempts as before." v2 doesn't write `actions` rows at all (audit-log events instead), and the discovery confirmed nothing outside the v1 orchestrator reads `retry_of_action_id` or `attempt_number` (no service, router, or UI consumes them). So "same number of attempts" becomes a behavioral assertion via audit events, not a row-count assertion.
- v1's "two critics over one draft" pattern is awkward today: voice and accuracy each have their own `CritiqueStep` with `critiques_step_index=4`, and on fail the orchestrator rewinds the workflow's `current_step`, writes a new draft row with `retry_of_action_id` chained back, and re-runs everything from there. The `_compute_attempt` / `_advance_or_retry` machinery in `app/orchestrator/prompt_chain.py` exists almost entirely to make this work. v2 expresses the same logic as a graph with conditional edges and shared state.

After Phase 3, only Phase 4 (multi-agent primitives) and Phase 5 (cleanup) remain. The bespoke orchestrator becomes deletable.

## Decisions (locked before drafting)

1. **Two critique loops share one `draft` node, not two.** Both voice and accuracy failures route back to the same `draft` node. State keeps independent `voice_attempts` / `accuracy_attempts` counters (matches v1 — each critic had its own budget; voice's max_attempts=3 and accuracy's max_attempts=2 are independent). When accuracy fails and we loop, voice runs again on the new draft. This matches v1 exactly.
2. **Outreach LLM calls go through `invoke_agent`.** `OutreachAgent`, `VoiceCriticAgent`, and `AccuracyCriticAgent` are all in `AGENTS` and all use Anthropic — exactly what `invoke_agent` supports. Complies with CLAUDE.md's "Never instantiate agent classes inside nodes."
3. **Content-creation LLM calls stay on direct `call_openai`.** `ContentStrategyAgent` and `LinkedInWritingAgent` aren't registered in `AGENTS`; `PersonalVoiceAgent` is, but all three use OpenAI and `invoke_agent` is Anthropic-only. Wrapping `invoke_agent` to dispatch by provider is Phase 4 territory. Document this gap; don't fix it here.
4. **One workflow_id per content_creation run.** No spawn, no looping outside the single graph — same shape as the critique POC.
5. **v1 coexistence: unregister immediately.** Same approach as Phases 1 and 2. Drop `OUTREACH_CHAIN` and `CONTENT_CREATION_CHAIN` from the v1 registry; leave the chain objects defined for Phase 5 deletion.
6. **Tests: skip v1, defer v2.** Same posture as Phase 2 ("code first, tests later" preference). `tests/test_outreach_workflow.py` gets a module-level skip; `tests/test_chain_diagram.py` parametrize lists drop both kinds. New v2 tests deferred to a single test-debt sweep alongside Phase 1's and Phase 2's pending tests.

## Graph topology — outreach

```
[entry] → pull_hubspot → web_search → consolidate → retrieve_kb → draft → voice_critique
                                                                    ▲          │
                                                                    │          ▼ (route_after_voice)
                                                                    │     ┌────┼─────────┐
                                                                    │   pass  fail+    fail+
                                                                    │    │   budget  exhausted
                                                                    │    │     │         │
                                                                    │    ▼     │         ▼
                                                                    │ accuracy_critique  failed_terminal → END
                                                                    │    │                  ▲
                                                                    │    ▼ (route_after_accuracy)
                                                                    │ ┌──┼──────────┐       │
                                                                    │ pass fail+    fail+   │
                                                                    │  │  budget  exhausted │
                                                                    │  │    │         └─────┘
                                                                    └──┘    │
                                                                    │       ▼
                                                                    │  propose_send
                                                                    │       │
                                                                    │  [interrupt_before gmail_send]
                                                                    │       │
                                                                    │       ▼ (after approve)
                                                                    │  gmail_send → END
                                                                    │
                                                              (loop back when either critic fails with budget)
```

`interrupt_before = ("gmail_send",)`

Two conditional edges: `route_after_voice` and `route_after_accuracy`. Both can route back to `draft` (loop) or forward to the next critic / approval gate, or to `failed_terminal`. Independent `voice_attempts` / `accuracy_attempts` counters in state — matches v1's per-CritiqueStep budget.

## Graph topology — content_creation

```
[entry] → interpret_brief → draft_post → voice_review
                                ▲           │
                                │           ▼ (route_after_voice)
                                │     ┌─────┼─────────┐
                                │   pass  fail+    fail+
                                │    │   budget  exhausted
                                │    │     │         │
                                │    ▼     │         ▼
                                │   END    │    failed_terminal → END
                                │          │
                                └──────────┘
                                  (loop)
```

No interrupt gate — `content_creation` terminates when voice passes (DB row → `status='ready'`) or budget exhausts (DB row stays `status='draft'`, workflow `failed`). Same shape as the critique POC.

## State design

### `OutreachState`

```python
class OutreachState(BaseGraphState, total=False):
    # From trigger
    hubspot_contact_id: NotRequired[str]
    notes: NotRequired[dict[str, Any]]

    # Built by upstream nodes
    contact: NotRequired[dict[str, Any]]
    company: NotRequired[dict[str, Any]]
    web_signals: NotRequired[dict[str, Any]]
    brief: NotRequired[str]
    gtm_blurb: NotRequired[str]

    # Current draft (overwritten each draft attempt)
    draft: NotRequired[dict[str, Any]]   # {to, to_name, subject, body}

    # Critique state (independent budgets; mirrors v1)
    voice_attempts: NotRequired[int]
    voice_max_attempts: NotRequired[int]      # default 3
    last_voice_critique: NotRequired[dict[str, Any]]

    accuracy_attempts: NotRequired[int]
    accuracy_max_attempts: NotRequired[int]   # default 2
    last_accuracy_critique: NotRequired[dict[str, Any]]

    # Most recent failed critique — surfaced in the next draft prompt
    last_critique_feedback: NotRequired[dict[str, Any]]

    # Approval bridge
    executed_payload: NotRequired[dict[str, Any]]

    # Terminal states
    result: NotRequired[dict[str, Any]]
    failure_reason: NotRequired[str]
```

## Implementation steps

### Step 1 — Define `app/orchestrator_v2/graphs/outreach.py`

Nine nodes mirroring v1's logic, restated as state-in/state-out functions. Concrete shapes (abbreviated; see v1 `outreach.py` for full bodies):

- **`pull_hubspot(state)`** — reads `state.hubspot_contact_id`. If `settings.hubspot_token` is set, raise `NotImplementedError` (matches v1). Otherwise return stub `{contact, company, hubspot_contact_id, stub: True}`.
- **`web_search(state)`** — stub `{web_signals: {company_name, signals: [...], stub: True}}`.
- **`consolidate(state)`** — `invoke_agent("outreach-agent", {"prompt": <brief prompt>, "max_tokens": 400}, ctx)`. Returns `{brief}`.
- **`retrieve_kb(state)`** — stub Frogslayer GTM blurb `{gtm_blurb, stub: True}`.
- **`draft(state)`** — `invoke_agent("outreach-agent", {"prompt": ..., "max_tokens": 600}, ctx)`. Builds the prompt; if `state.last_critique_feedback` is set, append the revision block (matches v1's `_draft_email`). Parses JSON via `parse_json`. Returns `{draft: {to, to_name, subject, body}, last_critique_feedback: None}` — explicitly clear the feedback after consumption.
- **`voice_critique(state)`** — load voice profile via direct SQL (same query as v1's `_load_voice_profile`). `invoke_agent("voice-critic", ...)`. Parse via `parse_json`. Increment `voice_attempts`. Set `last_voice_critique`; if not passed, also set `last_critique_feedback` so the next draft sees it.
- **`accuracy_critique(state)`** — `invoke_agent("accuracy-critic", ...)`. Same shape as voice_critique with the accuracy prompt. Increment `accuracy_attempts`. Set `last_accuracy_critique`; on fail, also set `last_critique_feedback`.
- **`propose_send(state)`** — return `{"_propose": ProposeApproval(action_type="send_email", agent_slug="outreach-agent", risk_level="medium", summary=<draft.subject or "Outreach email">, proposed_payload=state.draft)}`. Payload shape `{to, to_name, subject, body}` matches v1.
- **`gmail_send(state)`** — read `state.executed_payload` (or fall back to `state.draft`). Log a stub send. Return `{result: {stub: True, would_send_to, subject}}`.
- **`failed_terminal(state)`** — set `result = {"outcome": "failed", "reason": state.failure_reason or "critique budget exhausted"}` and route to END. Used when either critic budget is exhausted.

Routers:

```python
def route_after_voice(state) -> str:
    last = state.get("last_voice_critique") or {}
    if last.get("passed"):
        return "accuracy_critique"
    if state.get("voice_attempts", 0) >= state.get("voice_max_attempts", 3):
        return "failed_terminal"
    return "draft"  # loop

def route_after_accuracy(state) -> str:
    last = state.get("last_accuracy_critique") or {}
    if last.get("passed"):
        return "propose_send"
    if state.get("accuracy_attempts", 0) >= state.get("accuracy_max_attempts", 2):
        return "failed_terminal"
    return "draft"  # loop (will re-run voice too)
```

Wire it up:

```python
g.add_node("pull_hubspot", pull_hubspot)
g.add_node("web_search", web_search)
g.add_node("consolidate", consolidate)
g.add_node("retrieve_kb", retrieve_kb)
g.add_node("draft", draft)
g.add_node("voice_critique", voice_critique)
g.add_node("accuracy_critique", accuracy_critique)
g.add_node("propose_send", propose_send)
g.add_node("gmail_send", gmail_send)
g.add_node("failed_terminal", failed_terminal)

g.set_entry_point("pull_hubspot")
g.add_edge("pull_hubspot", "web_search")
g.add_edge("web_search", "consolidate")
g.add_edge("consolidate", "retrieve_kb")
g.add_edge("retrieve_kb", "draft")
g.add_edge("draft", "voice_critique")
g.add_conditional_edges("voice_critique", route_after_voice, {
    "accuracy_critique": "accuracy_critique", "draft": "draft", "failed_terminal": "failed_terminal",
})
g.add_conditional_edges("accuracy_critique", route_after_accuracy, {
    "propose_send": "propose_send", "draft": "draft", "failed_terminal": "failed_terminal",
})
g.add_edge("propose_send", "gmail_send")
g.add_edge("gmail_send", END)
g.add_edge("failed_terminal", END)

return GraphSpec(graph=g, interrupt_before=("gmail_send",))
```

### Step 2 — Define `app/orchestrator_v2/graphs/content_creation.py`

Four nodes: `interpret_brief`, `draft_post`, `voice_review`, `failed_terminal`. Mirror v1's logic, but call `call_openai` directly (the OpenAI/Anthropic split is documented; Phase 4 will unify via a provider-aware `invoke_agent`).

- **`interpret_brief(state)`** — `call_openai(ContentStrategyAgent.system_prompt, user_msg, model=ContentStrategyAgent.model)`. Parse with `parse_json`. Return `{idea}`.
- **`draft_post(state)`** — Build user msg from `state.idea`, `state.brief`, `state.channel`. If `state.last_voice_review` is set and didn't pass, append the revision block. `call_openai(LinkedInWritingAgent.system_prompt, ..., model=LinkedInWritingAgent.model, max_tokens=1000)`. Update the `social_posts` row via `update_post_conn` (status='draft', post_text, idea_title, core_angle). Return the draft as state. Same DB-side-effect contract as v1.
- **`voice_review(state)`** — `call_openai(PersonalVoiceAgent.get_system_prompt(channel), ...)`. Parse. If passed, `update_post_conn(post_id, status='ready', post_text=revised_or_original)`. Increment `voice_attempts`. Return `{voice_attempts, last_voice_review}`.
- **`failed_terminal(state)`** — set `result = {"outcome": "failed", "reason": "voice review budget exhausted"}`. The `social_posts` row stays at status='draft' (matches v1 code; v1 PROGRESS docs say "exhausted → needs_revision" but the code in v1 `content.py:_voice_review` only writes status on pass — open question below).

Router:

```python
def route_after_voice_review(state) -> str:
    last = state.get("last_voice_review") or {}
    if last.get("passed"):
        return "__end__"  # falls through to END
    if state.get("voice_attempts", 0) >= state.get("voice_max_attempts", 3):
        return "failed_terminal"
    return "draft_post"
```

Wire it up:

```python
g.add_node("interpret_brief", interpret_brief)
g.add_node("draft_post", draft_post)
g.add_node("voice_review", voice_review)
g.add_node("failed_terminal", failed_terminal)

g.set_entry_point("interpret_brief")
g.add_edge("interpret_brief", "draft_post")
g.add_edge("draft_post", "voice_review")
g.add_conditional_edges("voice_review", route_after_voice_review, {
    "draft_post": "draft_post", "failed_terminal": "failed_terminal", END: END,
})
g.add_edge("failed_terminal", END)

return GraphSpec(graph=g)  # no interrupts
```

> **Note on `social_posts.status` on terminal failure:** v1's `_voice_review` only updates the row when `passed=True` (sets status='ready'); on fail it leaves the row alone, and the workflow ends `failed` with the post still at status='draft'. v1's PROGRESS.md describes "exhausted → `needs_revision` + workflow `failed`" — but the code in `content.py:_voice_review` only writes status on pass. Open question to flag during execution: is the docs wrong or the code wrong? Phase 3 ports the **code's** behavior (status stays 'draft' on terminal failure). If we want the doc-described `needs_revision`, add one line to `failed_terminal` and update PROGRESS in the same commit.

## Risks & open watches

- **Critique-feedback prompt clarity.** v1 surfaces only the *most recent* failed critique in the redraft prompt. If voice fails twice and accuracy then fails, the next draft only sees the accuracy feedback, not the voice issues. v2 must mirror this — overwrite `last_critique_feedback` on each failure; clear it inside `draft` after consumption. Don't accidentally accumulate.
- **Cross-critic budget interaction.** Voice's budget is *per-voice-invocation*, not per-draft. If a draft passes voice on attempt 1 and fails accuracy, the loop returns to draft; voice runs again on the new draft, potentially failing — voice_attempts is now 2 even though it passed on attempt 1. This matches v1 (each `CritiqueStep` has its own attempt counter that increments every time it's reached). Document this in the node docstring; do **not** reset on loopback.
- **Infinite-loop guard.** None at framework level. The two budgets bound the loop in practice (voice 3 + accuracy 2 = at most ~5 drafts). Document in the node docstring.
- **Audit-event volume.** Each loop iteration emits node-entered/exited events. Workflows that hit budget produce ~5x the events of a single-pass run. Acceptable.
- **`AsyncPostgresSaver` checkpoint accumulation across loops.** Same caveat as Phase 2.
- **content_creation status semantics.** v1 docs say "exhausted → `needs_revision`"; v1 code only writes status on pass. Pick one and update both code + docs in the same commit. (See Step 2 note.)
- **Diagram endpoint test.** After Phase 3, the v1 chain registry is empty for production kinds. `tests/test_chain_diagram.py` may need a wholesale module-level skip rather than just dropping kinds — reassess during Step 6.
- **invoke_agent's Anthropic-only constraint.** Content_creation will not benefit from the audit events that `invoke_agent` emits (`AGENT_INVOKED` etc.). Acceptable for Phase 3; Phase 4's provider-aware refactor closes this.
