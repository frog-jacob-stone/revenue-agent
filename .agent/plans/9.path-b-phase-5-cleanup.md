# Phase 5 — Cleanup and Decommission v1

🔴 **Complex** — large surface area: 12 v1 orchestrator files deleted, 5 backend models/routers/services touched, 6 frontend files simplified, 4 v1 test files deleted, 2 schema migrations, a directory rename (`app/orchestrator_v2/` → `app/orchestrator/`), and a final docs pass. No new features — pure subtractive work that ends with v2 BEING the orchestrator.

Sub-plan for the master plan at `.agent/plans/3.langgraph-multi-agent-rearchitecture.md` (Phase 5 section).

## Context

After Phase 4, every production workflow runs on LangGraph (`app/orchestrator_v2/`), every cross-agent message is recorded in `agent_messages`, and the inbox UI dual-sources from `/actions` (v1) + `/approvals` (v2). The v1 surface is dead code that we keep around only for historical compatibility. Phase 5 deletes it.

Why now:
- All four v1 chains (`content_publish`, `rev_rec_monthly`, `outreach_chain`, `content_creation`) have v2 equivalents in production.
- The v1 `actions` table receives no new writes (every dispatch site has been flipped).
- Carrying both orchestrators imposes ongoing cognitive cost: every contributor has to know which surface a kind lives on, which API endpoint to use, which type to render in the UI.

After Phase 5: one orchestrator (`app/orchestrator/` = the LangGraph runner). One approval surface (`/approvals`). One inbox type (`Approval`). The bespoke prompt-chain machinery is gone.

## Decisions (locked before drafting)

1. **Delete v1 entirely; migrate nothing from it.** No v1 chain rebuilt. No v1 model preserved. Everything goes — `app/orchestrator/`, `app/routers/actions.py`, `app/services/execution.py`, `app/services/approval.py`, `app/models/actions.py`. The v2 surface owns 100% of orchestration after this phase.
2. **Module rename: `app/orchestrator_v2/` → `app/orchestrator/`.** v2 becomes THE orchestrator. The `_v2` suffix was only meaningful while two orchestrators coexisted. Mechanical find/replace through the codebase.
3. **`/chains` endpoint and UI page are dropped, not migrated.** The chain visualizer was built off v1's Step abstraction; rebuilding it on top of LangGraph's `StateGraph` (which already produces a Mermaid representation natively via `get_graph().draw_mermaid()`) is a separate piece of work. Phase 5 deletes the v1 implementation; revisualizer lands in a future ticket if useful.
4. **Pre-Phase-5 gate: drain in-flight v1 workflows.** Any `workflows` row with `status='awaiting_approval'` whose kind was a v1 chain (registered before Phases 1–3 unregistered them) becomes inaccessible after Phase 5. Operator action required: list and either approve via the existing v1 path before starting Phase 5 OR mark as `cancelled` directly in the DB. Plan calls this out explicitly as Step 0.
5. **Migrations 0014 and 0015, applied LAST.** Code changes happen first (drop the references); migrations drop the now-unreferenced columns and table. This sequencing means the test suite stays runnable throughout — at no point are we testing against a schema that doesn't match the code.
6. **Deferred Phase 1–4 test debt is NOT folded in.** Tasks #33 and #40 (the unwritten v2 graph tests) remain deferred. Phase 5's exit criterion is "the test suite that was green before Phase 5 is green after," not "every deferred test gets written." A separate post-Phase-5 sweep handles the test debt.
7. **Frontend changes are part of Phase 5, not deferred.** Removing the dual-source merge, dropping the `Action` type, simplifying `InboxList`/`InboxDetail`/`Sidebar`/`WorkflowTrace`. Without these, the UI references endpoints that no longer exist.

## Risks & open watches

- **In-flight v1 workflows lose their approval path.** Step 0 is the gate. If the operator skips it, those workflows are stranded. The plan calls this out; execution should pause at Step 0 for explicit confirmation.
- **`audit_log` foreign-key cascade.** `audit_log.action_id` references `actions.id`. Dropping `actions` with CASCADE will null the column or drop the rows. Need to verify whether the FK is `ON DELETE SET NULL` or `ON DELETE CASCADE` — if CASCADE, every audit row referencing an action gets deleted, which loses history. Pre-flight: check the migration that defined `audit_log.action_id` and add `ON DELETE SET NULL` first if needed before dropping `actions`. **Do this in migration 0014 as the first statement.**
- **`workflow_pattern` ENUM type at the SQL layer.** Migration 0005 likely created it. If so, `DROP TYPE workflow_pattern` is needed in 0015. Audit migration 0005 before writing 0015.
- **Module rename touches a lot of import sites.** A grep miss leaves a dead import that ImportErrors at boot. Step 4b's smoke check + Step 6's full pytest should catch all of them; if not, the verification step 6 catches stragglers.
- **Audit actor string change** (`"orchestrator_v2"` → `"orchestrator"`) breaks any saved query/dashboard that filtered on the old string. None known in the codebase; flag for downstream observability.
- **Frontend chains page deletion** removes a feature without replacement. If the user relied on it for visualization during the migration, communicate the loss; reimplementation is backlog.
- **Test file renames** are cosmetic and risk introducing inconsistency if done partially. Either rename all `test_orchestrator_v2_*.py` → `test_orchestrator_*.py` (cleaner) or rename none. Don't half-rename.
- **Frontend's `getInboxItems`** previously returned a discriminated union; consumers may have type narrowing on `isApproval(item)` that needs to be removed. TypeScript compiler will surface these.
- **Phase 5 is large.** Each step is independently small but the cumulative diff is significant. Resist scope creep — don't refactor anything not strictly required for the v1 deletion. Save renames-for-tidiness for after Phase 5 lands.
