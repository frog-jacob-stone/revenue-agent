# Phase 1 — Migrate `content_publish` to LangGraph (v2 runner)

⚠️ **Medium** — coordinated backend + UI change, but each piece is small and the v2 infrastructure (Phase 0) is already in place.

Sub-plan for the master plan at `.agent/plans/3.langgraph-multi-agent-rearchitecture.md` (Phase 1, lines 129–142).

> **Status (as executed):** code complete; doc updates landed; tests deferred. See `PROGRESS.md` "LangGraph Migration → Phase 1" for what's done and what's open.

## Context

Phase 0 built the v2 LangGraph runtime alongside the legacy orchestrator: `app/orchestrator_v2/runner.py` with register/start/resume, the `approvals` table, `app/routers/approvals.py`, the audit-event taxonomy, and a critique-loop POC. No real chain has migrated yet.

Phase 1 is the first real migration. We pick `content_publish` because it's the smallest chain — one ExecutionStep, one approval gate, no critique cycles, no branching — so it isolates "did the runner actually work" from "did we model the chain right." If this lands cleanly, the harder migrations (rev_rec branching, outreach critique loops) get the same scaffolding.

After this phase, `content_publish` runs on LangGraph end-to-end: the `publish_post` content-orchestrator tool dispatches to the v2 runner, the inbox surfaces the resulting approval row alongside v1 `actions` rows, and approving it advances the graph through to `social_posts.status = 'published'`.

## Decisions (answered before drafting)

1. **Inbox UI strategy:** frontend dual-source. `InboxList` queries both `/actions` and `/approvals` and merges client-side via a TypeScript union type. No new backend endpoint to retire in Phase 5.
2. **Checkpointer:** swap `MemorySaver` → `AsyncPostgresSaver` now. Adds migration 0012 for LangGraph's checkpoint tables. Derisks Phase 2's multi-interrupt rev_rec graph.
3. **v1 coexistence:** unregister the v1 `CONTENT_PUBLISH_CHAIN` immediately. The user has confirmed there's no production data to preserve; parity is verified against the documented end state (see Verification §3), not against a live-running v1.

## Implementation steps

### Step 1 — Migration 0012 (LangGraph checkpoint tables)

Create `supabase/migrations/0012_langgraph_checkpoint_tables.sql` as a marker file. The actual DDL is issued by `AsyncPostgresSaver.setup()` at app startup (idempotent). Marker keeps the migration ledger truthful without duplicating LangGraph-internal schema.

### Step 2 — Swap checkpointer in `runner.py`

Currently `runner.py:59`: `self._checkpointer = MemorySaver()`.

- `__init__`: `self._checkpointer: AsyncPostgresSaver | None = None`, `self._initialized = False`
- New `async def init(self)`: build `AsyncPostgresSaver` from a connection on `await get_pool()`, call `await checkpointer.setup()`, set `self._initialized = True`. Idempotent.
- `register()` stays sync; deferred compile happens lazily on first `start`/`resume` (compile needs the checkpointer)
- App startup: `await runner.init()` before `register_all(runner)` in `app/main.py` lifespan

> **As implemented:** AsyncPostgresSaver uses **psycopg**, not asyncpg. The checkpointer holds its own `psycopg_pool.AsyncConnectionPool` against `settings.database_url` — separate from the app's asyncpg pool. `register()` only stores the factory; compile happens lazily inside `_ensure_init()` on the first `start`/`resume`.

## Risks & open watches

- **AsyncPostgresSaver + conftest rollback.** Conftest does per-session rollback on `tests`-managed tables; LangGraph's checkpoint tables aren't in that list. Mitigation: extend conftest to truncate the three checkpoint tables between tests, or rely on unique `thread_id` per test (workflow_id is already a UUID, so this is automatic — no leak between tests, just accumulating rows). **AsyncPostgresSaver is psycopg-based, so its writes bypass the asyncpg test transaction wrapper entirely** — tests that pause a workflow then read state through the runner will see Postgres-backed state, not in-test-transaction state.
- **Frontend type union ergonomics.** `InboxItem = Action | Approval` means `.action_type` access still works (both have it) but anything else requires the discriminator. Keep the union narrow at the boundary; convert to a unified shape inside `getInboxItems` if it gets noisy.
- **Trace endpoint backward compat.** Adding `events` to `WorkflowTraceResponse` as optional doesn't break v1 consumers. UI gates on its presence.
- **v1 chain file still imported.** Unregister-only (don't delete the chain file in this phase). Phase 5 deletes it.
